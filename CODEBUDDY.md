# 基本农田保护监测平台 —— 全栈开发规范 Skill v2.0

本文件是团队成员与 AI 助手共同遵守的项目硬约束。违反核心架构、安全或 GIS 规则的代码不得交付。

可发现的 Codex Skill 位于 `.codex/skills/remote-sensing-gis-platform/`。涉及本项目的架构、GIS、前端布局或业务开发时，应使用 `$remote-sensing-gis-platform`，本文件与 Skill 必须同步维护。

## 1. 核心架构铁律

### 后端四层分离

后端调用方向固定为 `Routes → Services → DAO → Models`。

- Routes：只接收请求、通过 Pydantic 校验参数、调用 Service、返回响应；禁止调用 DAO 或编写 SQL。
- Services：只编排业务逻辑、调用 DAO、处理自定义业务异常；禁止处理 FastAPI `Request` 等 HTTP 对象。
- DAO：只负责 SQLAlchemy CRUD 和参数化空间查询；禁止抛出 `HTTPException`。
- Models：只定义 SQLAlchemy ORM 映射，禁止包含业务逻辑。

### 前端四层分离

前端调用方向固定为 `Views → Components → Store → API`。

- API 只能使用 `request.ts` 封装的 Axios 实例，组件中禁止直接使用 `axios` 或 `fetch`。
- `mapStore` 是地图视角和二三维状态的唯一真相来源。
- `layerStore` 是地图图层状态的唯一真相来源。
- 禁止通过 props 逐层传递 OpenLayers `map` 或 Cesium `viewer` 实例。
- 业务模块必须使用独立路由和独立 View；共享空间能力应抽取为组件或 Store，禁止堆入单个巨型 View。

## 2. 命名与代码风格

### Python

- 文件名使用小写下划线，例如 `plot_dao.py`。
- 类名使用大驼峰，例如 `FarmlandPlotDAO`。
- 函数和变量使用小写下划线。
- 所有函数参数和返回值必须显式标注类型。
- 每个类、公共方法必须包含中文说明；有参数和返回值时必须包含中文 `Args:`、`Returns:`。
- 复杂空间 SQL 上方必须说明业务含义。

### TypeScript / Vue

- Vue 组件文件使用大驼峰，Store 和工具文件使用小驼峰。
- 业务源码和工程配置统一使用 TypeScript，禁止新增 `.js` / `.jsx` 业务文件。
- 强制使用 `<script setup lang="ts">` 组合式 API。
- `ref` 变量使用 `Ref` 后缀，`computed` 变量使用 `Computed` 后缀。
- 复杂 OpenLayers、Cesium 坐标转换和相机控制必须注释说明。

## 3. 数据库与 SQL 安全红线

- 禁止使用 Python f-string 或字符串拼接 SQL。
- 查询必须使用 SQLAlchemy 表达式，或 `text()` 配合命名参数绑定。
- 内部空间坐标统一为 WGS84（EPSG:4326）。
- 空间函数两侧几何必须使用相同 SRID。
- 高频查询 `geom` 字段必须创建 GIST 索引。
- 表结构变更必须同步更新 `backend/scripts/init_db.sql`，或通过 Alembic 迁移。

## 4. API 与通信规范

- 成功响应统一为 `{"code": 200, "data": {...}}`。
- 失败响应统一包含 `code` 和 `msg`，同时使用正确 HTTP 状态码。
- 请求拦截器必须添加 `X-Requested-With: XMLHttpRequest`，Token 存在时统一注入。
- 前端遇到业务 `code !== 200` 必须通过全局 `message.error(msg)` 提示。
- 生产环境禁止向控制台输出后端原始异常堆栈。
- CORS 必须允许 `http://localhost:5173` 和 `http://localhost`。

## 5. 二三维 GIS 联动规范

- `mapStore` 是二三维视角唯一状态源。
- 2D 或 3D 主动交互后必须更新 `mapStore.center` 和 `mapStore.zoom`。
- 视图挂载时必须从 `mapStore` 恢复视角。
- Cesium API 经纬度使用 WGS84 度数值。
- OpenLayers GeoJSON 必须显式指定：

```typescript
new GeoJSON().readFeatures(data, {
  dataProjection: 'EPSG:4326',
  featureProjection: 'EPSG:3857',
})
```

- 三维视图不得自动旋转、自动飞行或持续移动，只允许用户主动拖拽、滚轮缩放和旋转。

## 6. 容器化与部署

- 后端和前端生产镜像都必须使用多阶段构建。
- 前端生产容器必须由 Nginx 提供静态文件，禁止运行 Vite 开发服务器。
- 密码、Token、数据库连接串禁止硬编码进业务代码或 Dockerfile。
- 本地 `.env` 必须被 `.gitignore` 忽略。
- PostGIS、后端、前端必须配置健康检查。
- 后端 `depends_on` 必须使用 `condition: service_healthy`。

## 7. Git 提交规范

- 创建提交前必须检查仓库当时真实生效的 commitlint、Git hooks 和相关 package scripts，不得根据经验臆造格式。
- 检查范围至少包括 `commitlint.config.*`、`.commitlintrc*`、各级 `package.json`、`.husky/`、`.git/hooks/`、`git config --get core.hooksPath` 和 `pre-commit` 配置（存在时）。
- 仓库实际配置优先于本文档示例；没有检测到 Conventional Commits 约束时，不得自行把 `<type>(<scope>): <subject>` 当成强制格式。
- 若仓库配置要求 Conventional Commits，可按其允许的 type 和 scope 编写；不得假定 type 列表固定不变。
- 提交标题和正文必须使用中文，内容详细、专业，并准确说明变更范围、关键行为、兼容性影响和验证结果；多模块或高风险变更不得只写笼统的单行标题。
- 未经用户明确授权，不得使用 `--no-verify` 绕过提交校验。

## 8. AI 辅助编程约束

- 禁止把超过 100 行的跨层业务代码堆入单个入口文件或单个巨型 View。
- `try-except` 必须捕获具体异常，禁止静默吞掉未知异常。
- Cesium 必须通过 `vite-plugin-cesium` 集成，禁止手工复制构建资源。
- 生成完整新模块时，先明确目录结构，再按文件实现。
- 每次变更后执行与风险匹配的静态检查、测试、构建和浏览器验证。

## 9. Vben5 前端布局与管理规范

前端应用壳层参考 Vben5 的架构思想，但不得整套复制 Vben Monorepo 或照搬其品牌皮肤。当前参考基线为 `vbenjs/vue-vben-admin` commit `0cd87c1`（2026-07-18）。

- 布局偏好必须由类型化 Pinia Store 管理并持久化，至少覆盖侧栏、工作区标题、多标签页、KeepAlive、内容宽度、信息密度、过渡和最大化状态。
- 侧栏、标签栏、内容路由、偏好抽屉必须拆分为独立布局组件，`WorkbenchLayout.vue` 只负责组合。
- 页面标题、描述、全宽模式、缓存和后续权限由路由 `meta` 驱动，禁止在多个组件重复维护页面配置。
- 多标签页必须支持固定首页、关闭当前/其他/全部、局部刷新、持久化和内容最大化；刷新不得使用整页浏览器重载。
- GIS 空间工作台强制保持全宽；居中内容模式仅应用于报表、配置、表单等非空间页面。
- 页面过渡只允许短时透明度/位移动画，严禁影响 Cesium 相机或触发地图自动定位。
- 界面设置应保存在浏览器本地，只影响展示偏好，不得改变业务数据和权限。
- 采用 Linear 式克制层级：紧凑间距、细边框、少阴影、单主色；采用 Sentry 式异常恢复：问题证据、严重度、动作和审计记录就近呈现。
- 所有交互必须覆盖加载、空、错误、禁用、无权限、未保存和破坏性确认状态。
- 导航徽标、总览卡片和模块数量必须来自后端当前项目/任务实时统计；禁止使用固定数字冒充影像资产、灾害斑块、外业疑点或审核待办。
- 顶部通知必须由当前任务真实阻断项和待办数量生成，并提供对应业务页面跳转；没有多项目切换能力时不得显示伪下拉项目选择器，服务状态不得永久写死为正常。
- 项目总览流程不得在前端预设“已完成”。影像、解译、质检、外业、三级审核和成果交付阶段必须由后端根据实体文件、任务作用域、质量门禁、待办状态和当前成果包证据计算；总体进度不得使用初始化固定百分比。

## 10. 特殊业务规则

- `plot_code` 是图斑唯一业务标识。

### 省级遥感生产与农业监测扩展基线

- 平台必须覆盖生产批次、任务分包、多源数据目录、多时相变化检测、独立监理、完整影像生产、专题制图、标准化归档、服务共享、涉密安全、无人机、田间物联网和病虫害预警，不得只建设单任务解译页面。
- 生产计划必须按影像到达批次和行政区生成可审计作业包，保存面积、图斑量、负责人、期限、进度、合并冲突和交付状态。
- 多时相变化检测必须绑定经过实体校验的前后时相影像、规则版本和基础数据版本，支持同步对比、卷帘/闪烁、候选变化、六类变化标注、排除原因、置信度和人工复核。
- 多时相变化检测固定复用 `change_detection_runs`、`change_candidates` 和 `change_detection_events`。检测任务必须按采集时间顺序绑定两期不同的 `imagery_assets` 和一个实体有效的 `imagery_registration_jobs` 配准成果，且参考景、待配准景和任务必须完全一致；要求业务影像、文件和 SHA256 有效、定标/校正完成、WGS84 范围相交，并使用物理输出复算且不超过项目规则阈值的残差。禁止再由前端填写配准方法、偏差或任意证据 URI；创建时固化规则版本、规则值、任务更新时间、显式任务图斑数、配准任务编号、输出 SHA256、残差、重叠率和相关质量证据。
- 每个检测任务只能原子导入一个由 1–500 个 EPSG:4326 Polygon 组成的冻结候选 FeatureCollection；任务更新时间或显式图斑数与创建快照不一致时必须拒绝并重新建任务。必须保存来源名称/URI/版本、来源要素 ID、内容 SHA256、批次编号、证据 URI、置信度、PostGIS 重算面积和六类变化编码，并使用检测任务冻结的 200/400 平方米规则快照校验。人工确认、重分类或排除必须校验稳定用户能力、填写证据并追加不可变事件；排除还必须填写原因。
- 内置自动候选发现只能读取检测任务两期实体栅格生成的同一公共网格预览。可选算法必须来自服务端注册表，每种算法必须具有独立编码、名称、版本、真实评分公式、说明、默认阈值和阈值范围；禁止只更换展示名称却复用同一计算结果。当前仅声明 `rgb_absolute_difference`（RGB 平均绝对差分）与 `rgb_change_vector`（RGB 变化向量幅值），不得冒充 NDVI、地类分类或业务变化结论。
- 自动候选成果必须保存算法编码、名称、版本、评分公式、全部参数、两期预览 SHA256、实体 GeoJSON URI 和成果 SHA256。自动结果只能写为 `unclassified`，人工归入六类变化编码并提供证据后才能确认；候选超过上限时必须明确拒绝并要求调整参数，禁止静默截断或把部分结果冒充完整批次。
- 自动候选连通域矢量化不得把仅角点相接的像元合并为自接触 Polygon。候选必须逐个通过 PostGIS 有效性、项目范围和冻结面积规则校验；任一无效几何或候选超限时整批拒绝，禁止落入部分候选或残留实体成果。
- 前后时相可视化必须由服务端读取检测任务绑定的两期实体栅格生成，不得用公共底图或浏览器独立重采样冒充。首次生成前重新核对源 SHA256，把两期影像重投影到同一 WGS84 公共交集网格并使用共同 RGB 拉伸，原子保存包含源/预览校验值和渲染器版本的缓存清单；前端卷帘、用户触发闪烁和并排模式必须共享同一缩放与平移变换。
- 政府采购量化规则基线包括：建设用地和设施农用地最小图斑 200 平方米、其他农用地 400 平方米、完整率不低于 98%、边界吻合度和地类准确率不低于 90%、关键字段准确率不低于 95%、影像套合偏差不超过 2 像素。规则必须按项目/任务配置并审计，禁止硬编码。
- 数据目录必须覆盖影像、矢量、表格、DEM、控制资料、气象、管理信息、无人机和物联网数据，保存来源、版本、SHA256、CRS、范围、时段、密级和派生血缘。
- 当前生产底座固定使用 `dataset_assets`、`dataset_lineages`、`production_batches`、`work_packages`、`work_package_plots` 和 `production_audit_events`。新增生产能力必须复用这些实体和 `/api/v1/production` 服务边界，禁止在前端另建本地批次或作业包状态。
- 数据资产必须区分“仅登记外部来源”和“上传实体并登记”。仅登记流程只能保存调用方校验值并进入 `pending`；上传实体必须由服务端限制大小、校验扩展名/签名/MIME、检查 ZIP/Office 路径穿越、加密和展开体积，计算大小与 SHA256 后原子发布到 `backend/storage/datasets`。同一项目相同 SHA256 不得重复登记，应建立派生或使用血缘。
- `verified` 数据资产必须同时保存受控 URI、原始文件名、大小、服务端 SHA256、媒体类型、核验时间、稳定用户编码、角色和至少 10 字依据。待核验或拒绝资产补传时必须锁定资产；SHA256 不一致要保存不可变 `rejected` 尝试且不得发布文件，数据库提交或回滚异常必须清理本次新文件和临时文件。
- 数据资产下载、生产批次时相选择和成果交付生成必须重新校验受控路径、格式签名、大小和 SHA256。生产批次只能绑定实体复核通过的 `imagery` 资产；交付目录必须保存已复核实体的受控 URI、大小、媒体类型、SHA256 和核验身份，不得只复制数据库状态。
- 多源数据目录批量入库固定为 1–20 个实体和一份逐文件清单，前端必须一次提交全部 `files` 与 `manifest_json`，禁止循环调用单文件接口冒充原子批次。清单必须保存批次编号、稳定操作人角色、至少 10 字依据、共同来源，以及逐文件顺序、文件名、资产编号、名称、类型、CRS、范围/时段、密级、业务状态和父资产血缘。
- 服务端必须先临时落盘并校验全部批次成员，文件名按大小写不敏感去重，同时拒绝重复资产编号、批内 SHA256、项目既有编号/SHA256、缺失父资产和批内循环血缘。全部通过后才排他发布，并在一个业务事务中写入资产、核验、血缘、批次、成员和生产审计；任何校验、发布、数据库、提交或回滚失败必须清除全部新目标文件和临时文件。
- 数据资产批次必须保存包含服务端实体大小、SHA256、媒体类型和格式检查结果的规范化 manifest JSON 及其 SHA256。生产总览必须返回真实批次计数和最近批次摘要；交付生成必须重新计算 manifest SHA256，并交叉校验成员数量、顺序、总大小、资产元数据、核验记录和物理实体状态后写入 `archive/dataset_import_batches.json`。
- 生产批次必须固化规则版本和完整规则快照。县区作业包必须由当前任务真实图斑生成，并逐条写入 `work_package_plots`；面积、图斑量和进度必须从显式关联聚合，禁止只保存县区名称或接受客户端汇总值。
- 生产批次与作业包创建、负责人/期限/状态、合并校核和交付状态变更必须写入 `production_audit_events`，保存稳定操作人编码、角色快照及修改前后值。
- 监理角色必须独立于内业和质检，支持抽样方案、过程检查、问题证据、整改期限、复检、县区评价和不可变监理报告。
- 独立监理固定复用 `supervision_plans`、`supervision_samples`、`supervision_inspections`、`supervision_findings`、`supervision_reinspections`、`supervision_county_evaluations`、`supervision_reports` 和 `supervision_events`。计划样本必须来自当前 `task_plots` 真实图斑，只保存身份、县区和版本快照，不得用固定编号或客户端样本替代；系统抽样和县区分层随机抽样必须可复现，单计划超过 5000 个样本时明确拒绝，禁止静默截断。
- 监理计划创建时必须固化任务图斑数量和任务更新时间，基础数据版本变化后拒绝继续追加检查或生成旧计划报告。计划、过程检查、问题登记、复检、县区评价和报告生成只允许启用的 `independent_supervisor`；整改证据必须由生产团队能力提交，独立监理不得代替被监理方整改。报告必须在至少一次过程检查、全部问题闭环、全部抽样县区完成评价后生成受控实体文件，并保存大小、SHA256、证据清单和稳定用户角色；下载前重新校验实体。
- 影像生产必须支持 RPC/GCP、DEM 正射、区域网平差、配准、融合、增强、匀色、镶嵌和县/省裁切；任何完成状态都必须有实体成果及分辨率、覆盖率、精度、波段、位深、格式和 SHA256 证据。
- GCP 几何精校正必须使用 3–100 个带真实来源的非共线控制点，校验像素范围、重复点、GCP/目标坐标系和重采样参数；服务端必须计算逐点像素残差、RMSE 和最大残差，超过显式门槛时拒绝生成成果。禁止信任客户端提交的残差或用普通重投影冒充 GCP 精校正。
- RPC 原始影像允许没有普通 CRS，但必须内嵌可解析 RPC 模型并由服务端推导 WGS84 范围；辐射和大气步骤必须继续传播 RPC 命名空间。RPC/DEM 正射只允许读取受控 DEM，校验其文件签名、CRS、实数高程、完整覆盖、大小和 SHA256，由 GDAL 服务端执行并在输出中保存 RPC/DEM 证据及 `ORTHORECTIFIED=true`；禁止信任客户端声明 RPC、DEM 校验值或正射完成状态。

- 外业疑点闭环及最终成果包生成前，每条外业记录至少具备一张通过实体校验的现场照片。成果包必须嵌入全部已验证外业实体、校验清单和事件账本，禁止将只有 URL 的记录描述为已纳入现场证据。
- 外业疑点页面必须完整提供“保留内业、采用外业、人工折中、驳回外业”四种决策和“重新打开”动作，禁止隐藏已支持路径或提交硬编码处置说明。每次处置必须填写人工复核依据；折中方案必须显式填写最终地类，最终为耕地时必须填写作物，禁止静默沿用内业属性。
- “采用外业”和“人工折中”只能修改已匹配且属于当前任务的图斑；未匹配记录不得虚构内业图斑修改。实际属性变化必须生成不可变图斑版本，并在审核记录中保存最终落地地类、作物、稳定用户编码和角色快照。
- 仅已处置疑点允许重新打开。重开必须恢复或新建外业质量问题、清空当前处置字段、将任务退回解译阶段并清空质量得分，同时保留上次结论并追加不可变重开审计，禁止删除历史。
- 影像处理步骤只有在 `backend/storage/imagery` 内存在受控格式实体文件，并通过格式签名、文件大小和 SHA256 校验后才能标记完成。数据库状态或虚构输出路径不得单独作为“已生成”依据。
- 影像入库必须从真实 GeoTIFF、IMG 或 HDF 文件读取驱动格式、CRS、WGS84 范围、分辨率、尺寸、波段和标签；文件使用受控目录持久化并建立 SHA256 去重，禁止在资产页使用硬编码数量冒充目录统计。
- 影像批量入库固定为 1–20 个文件和一份逐文件清单，必须在服务端一次请求、一次业务事务内完成。清单保存批次编号、稳定操作人角色、至少 10 字依据、文件顺序、原始文件名、资产编号、实体大小和 SHA256，并计算包含最终业务元数据来源审计的规范化清单 SHA256。文件名、资产编号、批内 SHA256 或数据库既有 SHA256 任一重复都必须整批拒绝；并发发布不得覆盖既有目标文件。
- 前端必须一次提交全部 `files` 和 `manifest_json`，逐文件展示并允许补录缺失的传感器、带时区采集时间、云量和处理级别；禁止循环调用单文件接口冒充原子批量导入。服务端必须先临时落盘并校验全部文件，全部通过后才发布实体和写入资产、默认处理步骤、批次、成员及审核记录；任一校验、发布或提交失败时不得留下数据库记录、已发布实体或临时文件。
- 从公共 STAC/COG 导入 Sentinel 等开放影像时，必须保存真实平台/载荷、STAC 条目、源波段 URL、产品 URI、处理基线、提供方、许可链接和公开密级；禁止把卫星型号写死、把公开影像标成涉密，或仅保存裁剪结果而丢失来源血缘。四个波段必须同 CRS、同仿射变换、同尺寸，并强制读取 STAC Raster Extension 的 scale/offset/nodata 转换为浮点反射率；缺少标度时整次导入失败，禁止把量化 DN 冒充 L2A 反射率。
- 公开历史影像检索必须由服务端固定提供方和 collection。前端只能提交有上限的 WGS84 范围、日期、云量和选中 Item ID，禁止提交或保存 COG/SAS URL。既允许一景真实足迹完整覆盖，也允许 2–10 景跨轨候选联合覆盖；导入时服务端必须重新读取全部 Item，使用 PostGIS 对真实 STAC Polygon/MultiPolygon 足迹执行逐景和 `ST_UnaryUnion` 联合覆盖校核，禁止用 bbox 并集冒充完整覆盖。覆盖通过后才校验无签名 HTTPS 来源主机、仅在内存中申请短期签名，每景裁取自身与共同目标范围的交集，并要求 Blue/Green/Red/NIR 四个地表反射率波段同网格，执行 scale/offset/nodata、像元上限和统一影像原子入库门禁。物理实体必须保存查询/裁取范围、逐景/联合覆盖率、景数和覆盖依据，并保留无签名 STAC/产品/WRS/提供方/许可/公开分类和“非法定调查成果”说明；子区或多景批次不得冒充完整 1980–2024 历史库。
- 公开历史影像原子批次固定为 1–10 个唯一 Item ID、一个共同 WGS84 bbox、唯一资产编号、一个批次编号、稳定操作人和至少 10 字依据。服务端必须先重新获取、签名并裁取全部条目，再且仅调用一次统一 `upload_assets_batch`；文件名使用 Item ID 稳定生成并保持请求顺序。任何网络、签名、栅格、业务、数据库或提交失败都必须清除全部公开临时子集，不得循环调用单景接口或保留部分成功。前端必须保留逐景资产编号/名称草稿，并明确展示最多十景、权限、加载、空态、错误和整批成功证据。
- 影像业务元数据必须优先从大小写无关的文件标签提取：传感器支持平台/载荷别名，采集时间支持常见卫星日期时间别名，处理级别和云量支持主流产品字段。文件缺失时才允许人工补录；文件值与人工值冲突必须拒绝入库，禁止静默覆盖。最终值、标签名、文件原值、人工值、时间精度和时区假设必须保存到 `raster_metadata.business_metadata`。
- 初始化和工作台不得依赖无实体文件的固定影像元数据。没有满足 `operational`、实体 URI、文件大小和 SHA256 条件的业务影像时，接口必须返回明确空值、前端显示空状态、质量检查保持影像缺失门禁；`demo` 影像必须持续显式标记。
- 影像入库表单不得预填卫星传感器、采集时间、处理级别等业务事实；解译、批量赋值和分割依据不得使用固定卫星名称或日期充当示例结论。初始化脚本不得创建固定人员、固定时间的伪审核记录。
- 平台内置影像处理必须使用明确且可审计的算法参数：比例/偏移定标、DOS1、真实 CRS 重投影、数据库行政区边界裁剪和显式波段映射。禁止将文件复制、预览渲染或未配置的 6S/FLAASH 冒充处理完成。
- 源产品级别承认必须按产品族独立校验，禁止只信任客户端填写的 L2/L2A。Sentinel-2 必须证明 MSI L2A、STAC 标度已应用和浮点 BOA 反射率；Landsat 必须证明受控 Collection 2 Level-2 导入血缘、Planetary Computer/USGS 提供方、产品与 STAC Item 一致、WRS、四个有序浮点 Blue/Green/Red/NIR 波段、完整有限标度清单、无签名 Azure COG URL 和公开许可。承认必须由具备 `process_imagery` 能力的稳定用户明确确认，保存产品族、处理器版本、来源证据、人工依据和 `algorithm_executed=false`，复用同一实体而不复制文件。几何校正、裁剪、增强和波段产品不得通过该机制跳过。
- 多景源产品级别承认固定为 1–10 个同一任务项目的唯一 `operational` L2A/L2 资产，每景必须同时承认辐射定标与大气校正两个步骤。请求必须包含稳定操作人、至少 10 字依据和明确“未执行算法”确认。服务端按资产编号稳定顺序锁定资产与双步骤，先完成全部受控路径、实体大小、SHA256 和产品族证据预检，再生成一个承认编号写入所有步骤；每步骤一条不可变事件并追加一条批次事件，整个批次只提交一次。任一成员或事务失败时所有资产保持原状态，不得产生部分审核记录或复制文件。
- 多景内置处理固定为 1–10 个同一任务项目的唯一 `operational` 影像和一个统一步骤。服务端必须按资产编号稳定锁定资产及其全部流水线，在生成成果前复核上游必选步骤、受控实体、大小、SHA256、步骤依赖、算法参数和真实行政区裁剪边界证据。批量几何处理只允许统一坐标系重投影，GCP 和 RPC/DEM 继续使用单景流程；只允许批量处理尚无有效产物的步骤，已完成步骤必须通过单景流程显式重跑。全部独立 GeoTIFF 成功后才写入批次编号、实际参数、源/输出证据、逐景事件和一个批次事件，并只提交一次。任一处理、持久化或提交失败必须删除本批全部新产物、清理 Rasterio 临时侧车并回滚全部状态。
- 真彩色、假彩色和 NDVI 的红、绿、蓝、近红外必须映射到四个不同的实体波段，前端应优先按波段描述识别。NDVI 只对有限、非负且分母大于零的反射率计算，无效像元使用 nodata，结果限制在 `[-1, 1]`。
- 影像处理产物必须临时写入后原子替换，并记录源/输出 SHA256、执行参数、栅格结构、处理器版本和稳定用户编码。重跑上游步骤必须自动失效全部下游产物，同时保留旧证据历史。
- 影像工作台不得用固定瓦片、公共底图或 CSS 滤镜冒充当前资产及真彩色、假彩色、NDVI 处理成果。源影像快视图只能读取当前实体资产并重新核验文件大小和 SHA256；专题产品快视图只能读取已校验的 `band_products` 实体产物及明确波段描述。缓存必须按来源 SHA256 隔离并保存来源 URI/大小、波段、WGS84 范围、拉伸或值域、渲染器版本、PNG 大小和 PNG SHA256；缓存读取需复核校验值。快视图仅用于显示核验，禁止改变处理状态或计入完成率。
- 影像工作台首次进入时必须等待当前项目总览与资产目录同步完成，并优先选择总览指定或已校验的 `operational` 业务影像；演示资产即使采集时间更新、波段更多或已有完整处理链，也不得抢占真实业务影像的默认选择。
- 历史影像覆盖矩阵必须使用数据库完整 13 个地级区域和 122 个县区真实边界，以县区 geography 面积为分母、影像 WGS84 足迹交集为分子，保留所有零覆盖单元并区分 `operational` 与 `demo`；禁止用包围盒面积、固定年份或前端预设覆盖率冒充历史覆盖。
- 影像问题追溯必须从实际入库时间、必选处理步骤、当前实体 SHA256 校验和 `artifact_history` 历史证据生成。源文件或步骤产物失效、云量超限和必选步骤待办必须明确显示；没有真实历史时相时返回空状态，禁止补造 1980–2024 固定时间线。
- 任务质量门禁必须通过 `task_plots` 明确限定任务图斑作用域，禁止把全库所有有效图斑默认当作任意任务的检查范围。新建或导入图斑必须同步建立任务关联。
- 任务底块快照重载必须在同一事务中删除旧 `plot_quality_checks` 当前态、审计关闭旧 `source=auto`/`issue_type=quality_rule` 问题并清空任务质量得分；人工审核问题、外业问题和历史审核日志必须保留。每次快照重载必须追加新的不可变导入审计，禁止覆盖旧记录。
- 全量质量检查必须使用批量空间指标查询和单事务持久化，禁止对数百个图斑逐条提交事务。重跑质检只能关闭 `quality_rule` 类型的旧自动问题，不得误关闭外业核查问题。
- 每次任务全量质检必须在同一事务保存不可变批次账本，至少包含质检编号、任务图斑数量与更新时间快照、项目规则版本及阈值快照、实际参与覆盖判定的最新业务影像编号/采集时间/受控 URI/大小/分辨率/SHA256 快照及其规范化摘要、活动自定义字段定义与规范化 SHA256、覆盖/通过/失败数量、平均得分、问题数、提交结论、耗时、逐规则汇总、稳定操作人角色和人工说明。前端必须展示有上限的历史账本，最终成果包必须内嵌 `quality/quality_runs.json`；禁止只依赖瞬时响应或自由文本审核记录证明全量质检已经执行。提交内业自检必须锁定并绑定最近一次通过的全量批次，复核任务时间、图斑/检查/通过数量、得分和影像摘要仍与当前证据一致，并把批次编号作为结构化审核字段保存。项目规则或活动字段模式变化必须清理当前检查、仅关闭旧自动规则问题并重开任务，但不得删除历史批次。新入库的 `operational` 影像只有在实际成为最新质量判定影像时，才可在影像入库同一事务内清理当前检查、以稳定用户证据关闭开放的自动规则问题、重开项目全部任务并要求重跑全量质检；演示影像和更旧业务影像不得误触发失效。

- 地块属性 Excel 必须由服务端从当前任务真实图斑生成，固定包含 `plot_code`、`expected_version`、权属村、地类、作物、种植模式和灌排条件；任务全量或显式范围单次不得超过 500 个图斑。禁止浏览器自行拼装数据库导出结果。

- 审核页面默认只能将最近一次数据重载、审核退回/驳回、版本回退或外业批次导入后的记录作为“当前整改周期”证据；完整历史必须保留用于成果包和审计总数，不得与当前待审依据混淆，也不得无界加载到工作台概览。

- 成果包列表和生成接口必须重新校验业务影像、任务质量检查全覆盖、全部图斑门禁通过和平均质量得分，不得只信任任务 `completed` 状态。质量/验收报告必须区分“有记录”和“空集合”；外业或灾害数据为 0 时必须明确写未提供，不得声称对应专项检查或成果已经完成。

- 源栅格离线介质封存必须以当前通过实体和归档快照复核的成果包为前置门禁，并完整纳入全部 `operational` 业务源影像、真正产生独立物理输出的已完成处理步骤，以及任务级/项目级已验证多源数据实体。源级承认步骤若复用源栅格不得重复写入介质；外部 URL、待核验登记、演示影像和失效产物不得冒充已封存实体。

- 离线封存必须先在同文件系统临时目录完成全部卷、顶层清单和逐成员复核，再原子发布；数据库写入或提交失败必须清理全部新文件和目录。封存版本、分卷、来源实体和生成/替代/下载事件固定持久化，保存稳定用户编码及角色快照。项目负责人生成，项目负责人和甲方下载；新版本替代旧完成版本，当前成果包或任一来源 URI、大小、SHA256、版本或更新时间变化后旧封存必须禁止下载。

- 作物长势监测只能选择两个不同采集时相、`operational` 且 `band_products` 实体大小和 SHA256 均有效的影像。基准期必须早于监测期，实体必须包含唯一明确描述为 `NDVI` 的波段；禁止使用固定 NDVI、公共底图、CSS 滤镜、客户端数组或用户填写的分类结果冒充长势计算。

- 长势分级阈值必须由请求显式提供并随任务固化，输出“差/正常/好”三类受控 GeoTIFF。转差区域按 4 邻域矢量化，再由 PostGIS 裁切到任务耕地、重算 geography 面积并应用最小面积门槛；单任务异常区超过 500 个时明确拒绝，禁止静默截断。
- 每个长势任务必须保存两期资产与步骤 URI/大小/SHA256、采集时间、NDVI 波段、任务耕地图斑数与更新时间、完整任务面积、共同足迹覆盖面积、空间覆盖率、有效像元率、两个独立门槛、分类统计、算法版本、稳定用户角色和生成依据。分级 GeoTIFF 与异常区 GeoJSON 必须原子发布；下载和成果归档前重新校验成果结构、大小、SHA256、异常区数量以及两期来源步骤 URI/大小/SHA256/NDVI 波段。新增任务或任一来源步骤更新必须使依赖旧归档快照的成果包失效。
