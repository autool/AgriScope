# Remote Sensing GIS Platform

Build production-oriented increments for the agricultural remote-sensing platform while preserving its architecture, GIS invariants, workflow auditability, and runnable state.

## Required workflow

1. Read repository instructions, `CODEBUDDY.md`, current source, runtime state, and tests before relying on earlier context.
2. Derive the complete user-visible workflow, database changes, API contract, frontend state, error states, permissions, and verification evidence required by the request.
3. Keep backend flow `Routes → Services → DAO → Models` and frontend flow `Views → Components → Store → API`.
4. Implement the real business behavior. Do not leave enabled controls that only display a message or use synthetic GIS data without an explicit demo label.
5. Preserve TypeScript strictness, parameterized SQL, WGS84 API geometry, explicit OpenLayers projection, and stationary Cesium behavior.
6. Verify in proportion to risk: backend tests and Ruff, frontend typecheck/lint/build, database queries, API calls, and browser screenshots or interaction checks.
7. Keep services runnable after changes when the task involves implementation.
8. Before committing, inspect the repository's currently effective commitlint, package scripts, and Git hook configuration. Follow the discovered rules, never invent a format, write the title and body in detailed professional Chinese, and never bypass hooks without explicit user authorization.

## Architecture gates

- Reject route handlers that contain SQL or call a DAO directly.
- Reject components that call Axios/fetch directly or receive OpenLayers/Cesium instances through prop chains.
- Reject JavaScript application source; use `.ts` and `<script setup lang="ts">`.
- Reject giant views that combine unrelated modules or duplicate business panels.
- Reject interpolated SQL, untracked schema edits, fake successful responses, silent exception swallowing, or raw backend stack traces in the browser.
- Reject automatic Cesium rotation, flight, camera drift, or movement caused only by switching to 3D. Explicit user navigation may reposition once without animation.
- Reject administrative or business boundaries drawn as placeholder rectangles when the feature is presented as real data.
- Reject flat administrative lists or plot-derived region trees that hide real empty regions. Province-wide resources must preserve province → prefecture → county work area → real parcel hierarchy.
- Reject province-wide OSM parcel snapshots below all 13 prefecture-level regions, all 122 county-level regions, 35,000 traceable Polygon features, or 110 county-level regions with at least 20 features each unless the documented business scope is explicitly reduced. Preserve valid 0.05–500 hectare closed parcels, including regular rectangles and traceable relation MultiPolygon parts.
- Reject OSM imports that flatten unlike land-use tags into one fake class. Explicitly map supported tags to farmland, orchard, forest, grassland, water, and construction land; preserve way/relation source IDs, relation part numbers, source links, versions, and timestamps; differentiate all six business classes in the catalog plus 2D/3D rendering.
- Reject hierarchy that exists only in data but is visually hidden behind collapsed or undifferentiated nodes. Province-wide workbenches must explicitly present province → prefecture → county work area → real parcel counts, group all 122 county entries under their 13 visible prefecture parents, expand every populated county through individual parcels by default, open the boundary catalog to county level, provide explicit prefecture/county/parcel expansion controls, and support region or plot search. Expanding only one representative branch or showing prefectures without their county children is not acceptable.
- Reject administrative boundaries that use visually indistinguishable strokes. The map must differentiate province, prefecture, and county boundaries by width and dash pattern, with an adjacent legend and live prefecture/county/plot coverage counts.
- At province scale, label real prefecture boundaries with their parcel counts so small land-use polygons are not mistaken for missing data; progressively reveal county work areas and individual parcel detail at closer scales.
- Reject province-wide workbenches that download every full parcel geometry to construct the catalog. Keep the complete task-scoped catalog as lightweight plot identity/source/class/bbox data, and load full Polygon geometry by task plus current WGS84 viewport only at editing scales. When a viewport exceeds 5,000 matches, return the exact count and an explicit zoom requirement with zero partial features; never present an arbitrary truncated subset as complete.
- Reject plot read paths that query the global plot table and defer task validation to later mutations. Point/click lookup, plot-code boundary lookup, workbench attributes, version history, and rollback target reads must carry `task_code`, verify `task_plots` before exposing the object, and return a task-scoped 404 for unassigned plots without revealing cross-task existence.
- Reject hard-coded quality or field-verification thresholds when the settings UI presents them as configurable. Persist project rules and audit every change.
- Reject browser-only XLSX parsing or partial field-record imports. Field CSV/XLSX batches are limited to 500 records and must reuse one server-side atomic matching transaction. Server-side XLSX parsing must validate ZIP structure, paths, encryption, expanded size, formulas, headers, timezone-aware capture times, WGS84 coordinates, and photo evidence while persisting the physical file SHA-256 plus stable uploader-role audit.
- Reject field workflows that provide only desktop CSV/XLSX import. Provide a standalone mobile-width capture route that records terminal WGS84 coordinates, reported horizontal accuracy, timezone-aware capture time, stable field-inspector identity, observed land/crop values, remarks, and a camera-selected physical photo. Require HTTPS outside localhost for browser geolocation, run authoritative project containment and task-plot matching on the server, and never describe an unuploaded photo as offline evidence.
- Make mobile retries idempotent. Return an existing field record only when task, investigator and canonical payload SHA-256 match; reject conflicting reuse of the verification code. Return an existing artifact for the same record/photo SHA-256 without creating a second file or upload event. If record creation succeeds but photo upload fails, keep an explicit resumable pending-photo state rather than claiming capture completion.
- Reject field photos, voice notes, or investigation forms represented only by caller-supplied URLs. Preserve legacy links as explicitly unverified references, but require controlled physical JPEG/PNG/WebP, WAV/MP3/M4A/OGG, PDF/XLSX artifacts with server-enforced size limits, signature/MIME validation, atomic storage, size and SHA-256, stable uploader-role events, and download-time revalidation. - Reject field-resolution interfaces that hide supported decisions or submit hard-coded evidence comments. Expose keep-internal, use-field, explicit compromise, reject-field, and reopen actions; require human evidence text, require final land/crop attributes for compromise, disable plot-changing decisions for unmatched records, version every actual plot attribute change, and preserve stable resolver-role audit. Reopening must restore the field quality issue and interpretation gate without deleting the prior resolution history.

In one transaction, remove derived `plot_quality_checks`, resolve only open automatic `quality_rule` issues with stable system evidence, reset the task score, preserve manual/field issues and full history, and append a new immutable import-cycle record.
- Re-running automatic quality checks may resolve prior `quality_rule` findings only; it must preserve field-verification and manually raised issues.
- Reject quality dashboards that expose counts without an actionable issue queue. Persisted findings must be pageable, filterable by rule/status/severity/region, and able to select and focus the associated plot on the map.
- Reject bulk attribute updates that infer crops or silently target a whole region. Bulk remediation must use explicit plot selections, task-scope validation, a required evidence comment, one immutable version per plot, and a quality recheck gate.

Recheck source SHA-256 before first generation, reproject both rasters to one WGS84 intersection grid, use the same RGB stretch ranges, persist an atomic cache manifest with source/preview checksums and renderer version, and apply one shared frontend zoom/pan transform to swipe, user-triggered flicker, and side-by-side modes.
- Reject generic asset lists that omit source lineage. Every imagery, raster, vector, tabular, control, DEM, weather, management, UAV, and IoT dataset must preserve source, version, checksum, CRS, extent, time range, security classification, task use, and derived-artifact lineage.
- Reject dataset catalog entries marked verified from caller-supplied checksums alone. External-source registration remains pending; entity upload must enforce size, extension/signature/MIME and safe ZIP/Office structure, compute SHA-256 server-side, publish atomically under controlled dataset storage, and persist stable verifier-role evidence plus a justification of at least 10 characters. A checksum mismatch creates an immutable rejected attempt without publishing the file, and transaction failure removes every new temporary or final file.
- Reject sequential single-file requests presented as a multi-source dataset batch. Accept 1–20 physical files and one itemized manifest in one multipart request and one business transaction. Validate every file before publication; deduplicate case-insensitive filenames, asset codes, in-batch SHA-256 values, existing project codes/checksums, missing parents, and cyclic in-batch lineage. Publish without overwriting concurrent targets; persist ordered assets, verifications, lineage, batch/member rows, stable operator-role evidence, a justification of at least 10 characters, canonical manifest JSON and its SHA-256. Any validation, publication, database, commit, or rollback failure must leave no new rows, published files, or temporary files.
- Revalidate multi-source dataset batch manifest SHA-256, member count/order/total size, asset metadata, verification records, and controlled physical entities before final delivery. Include the verified ledger as `archive/dataset_import_batches.json`; the production overview and TypeScript catalog UI must expose real batch counts and recent manifest summaries, while failed submissions keep unsaved item metadata available for correction.
- Revalidate controlled path, format, size, and SHA-256 before dataset download, production-batch use, or delivery generation. Production source/target slots accept only physically revalidated imagery datasets. Delivery catalogs preserve controlled URI, original filename, media type, size, SHA-256, verifier identity/role and verification basis, and distinguish verified references from pending external registrations.
- Reject a quality-inspector review being presented as independent project supervision. Supervision requires a separate role, sampling plan, process inspection, evidence, findings, rework deadline, reinspection, county evaluation, and immutable report.
- Reuse `supervision_plans`, `supervision_samples`, `supervision_inspections`, `supervision_findings`, `supervision_reinspections`, `supervision_county_evaluations`, `supervision_reports`, and `supervision_events` for independent supervision. Samples must come from the current task's explicit `task_plots` and preserve plot identity, county, plot version, task plot count, and task update time. Systematic and county-stratified random sampling must be reproducible; reject plans above 5,000 samples instead of silently truncating them or accepting browser-defined samples.
- Only an active persisted `independent_supervisor` may create plans, inspections, findings, reinspections, county evaluations, or reports. Rectification evidence must be submitted through a separate production-team capability so the supervisor cannot impersonate the supervised party. Reject further evidence or report generation after the task data snapshot changes. Generate a controlled physical report only after at least one inspection, every finding is closed, and every sampled county is evaluated; persist file size, SHA-256, evidence manifest, stable role evidence, and revalidate the artifact before download.
- Reject orthorectification, registration, fusion, mosaicking, color balancing, or thematic-map generation claims without checksum-backed physical artifacts and measurable resolution, coverage, positional, band/bit-depth, layout, and format acceptance evidence.
- Reject thematic maps rendered from quicklook caches, public basemaps, or unchecked paths. Read only a checksum-verified `band_products` artifact; persist the source URI/SHA-256, STAC/license/classification/baseline lineage, actual bands, stretch or value range, complete layout, renderer version, output size/SHA-256, and stable user-role audit. Revalidate controlled path, signature, size, and checksum before preview or download, and keep a visible non-statutory label on public-data maps.
- Reject thematic atlases that omit any current verified PNG thematic map, duplicate pages, accept browser-only previews, or concatenate unchecked files. Let users order the complete current set, then generate the cover, paginated table of contents, and normalized map pages on the server. Atomically publish one ZIP containing the physical PDF, original PNG members, and canonical manifest; persist atlas version, ordered member snapshots, source-set count/latest-time/digest, PDF page count, ZIP/PDF/member sizes and SHA-256 values, and stable user-role audit. A new atlas supersedes the prior completed version; any thematic-map addition, loss, corruption, or source-set change makes the old atlas non-downloadable. Revalidate the current source set plus ZIP/PDF/manifest/member evidence before download or delivery embedding, and include atlas count/latest time in delivery staleness.
- Reject public map/data services without persisted registration, independent approval, database-verified resource identity/SHA-256, documentation, classification, exposure scope, access requests, credentials, real health checks, usage audit, and revocation. Never trust a caller-supplied checksum for an internal resource, expose confidential data publicly, store API-key plaintext, or probe loopback/private/link-local/reserved health targets without an explicit deployment allowlist. Return a generated secret only once, store its SHA-256 and last four characters, authenticate usage records, and atomically revoke every active credential when a service is revoked.
- Reject UAV or monitoring-device records without real source identity, coordinates, capture/telemetry time, operator/device ownership, physical evidence, checksum, status, and maintenance/fault audit.
- Reuse `uav_aircraft`, `uav_missions`, `uav_artifacts`, `uav_findings`, and `uav_events` for UAV work. Require controlled aircraft-certificate and pilot-license files with server-computed size/SHA-256; validate each WGS84 flight Polygon is fully covered by the declared persisted district boundary. Store raw imagery, flight logs, photos, videos, orthomosaics, DEMs, and reports as physical artifacts; extract raster CRS/resolution/dimensions/footprint from the file, require orthomosaic coverage and target resolution, and revalidate controlled path/size/SHA-256 before download.
- Reject UAV status transitions that trust client completion flags. Capture completion requires verified raw imagery plus a flight log; processing completion requires a qualifying orthomosaic; review completion requires zero pending findings. Finding coordinates must lie inside the mission boundary, linked plots must belong to the mission task, and every registration, upload, transition, review, and download authorization must preserve stable user capability and immutable role-snapshot evidence.
- Reject UAV workbenches that make pasted GeoJSON or typed longitude/latitude the only operational path. Provide a stationary OpenLayers planning map with the current imagery tiles, real province/prefecture/county boundaries, mission Polygons, status-aware findings, direct WGS84 boundary drawing, and finding-coordinate picking. Treat every browser geometry as an untrusted draft and keep authoritative district containment, mission containment, and task-plot validation on the server.
- Reject pest/disease dashboards based on fixed alerts or unaudited model output. Preserve station/device observations, model code/version, inputs, confidence, validation metrics, prediction basis, human review, warning delivery, and superseded model evidence.
- Reuse `monitoring_stations`, `monitoring_devices`, `device_telemetry`, `device_faults`, `pest_model_versions`, `pest_assessments`, `pest_alerts`, and `monitoring_events` for field monitoring. Validate station coordinates against the declared persisted district boundary; require physical size/SHA-256 evidence for device photos, telemetry files, model artifacts, assessment inputs, fault resolution, and alert delivery. Device/idempotency-key duplicates may return the original row only when the canonical payload SHA-256 matches; conflicting payloads must fail. Supersede prior active model versions without deleting them, and forbid alerts before stable-user human approval.
- Reuse `pest_reports`, `pest_report_items`, and `expert_consultations` for pest/disease reporting. Build a report only from explicitly selected, human-approved assessments within its persisted time period and real administrative scope. Open consultations must block submission; answers require a physical PDF/Office/image/ZIP artifact with server-computed size and SHA-256. County, prefecture, and province review transitions must be server-authorized by stable user capabilities and recorded as immutable events; never let the frontend declare approval or completion.
- Reject pest/disease reports represented only by database status, browser-generated tables, empty files, or unchecked download links. Province approval must atomically produce a physical XLSX ledger containing the report summary, assessment evidence, review trail, and consultation evidence; persist its size and SHA-256 and revalidate the controlled path, size, and checksum before download.

## Frontend product principles

- Adapt Vben5 patterns; do not import the entire framework or copy its visual identity.
- Drive the application shell from typed, persisted preferences: sidebar, header, route tabs, KeepAlive, content width, density, transition, and maximize state.
- Drive menu and page behavior from route metadata. Keep business pages independently lazy-loaded.
- Optimize desktop GIS work for scan speed, split panes, queues, filters, maps, detail panels, audit history, and recovery states.
- Use restrained color and hairline surfaces. Reserve strong color for selection, severity, status, and primary actions.
- Define loading, empty, error, disabled, permission-denied, offline, unsaved, and destructive states for interactive features.
